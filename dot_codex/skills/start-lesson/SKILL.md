LESSON.md を読み込み、対話形式のレッスンを開始する。以下の手順で進める:

1. カレントディレクトリの `LESSON.md` を読み込む。存在しない場合は「LESSON.md が見つかりません。プロジェクトリードに `/generate-lesson` で生成してもらってください。」と案内して終了する。

2. `LESSON_FEEDBACK.md` が存在するか確認する。存在すれば読み込み、既存のフィードバックを把握する。

3. LESSON.md の内容（問題文）を把握する。領域構成（共通 / フロントエンド / バックエンド 等）を認識する。

4. 以下のメッセージを表示してレッスンを開始する:

```
## Project Lesson を開始します

LESSON.md から N 問のレッスンを読み込みました。

### 領域一覧

| 領域 | 問題数 | 概要 |
|---|---|---|
| 共通 | x 問 | プロジェクト全体の構造・規約 |
| <領域1> | x 問 | ... |
| <領域2> | x 問 | ... |

### 進め方

- **まず「共通」から**: 全員が取り組むことを推奨
- **次に担当領域**: 自分の担当に合った領域を選んでください
- 全領域やらなくても OK。担当外の領域は余裕があるときに

### コマンド

| コマンド | 例 | 説明 |
|---|---|---|
| 回答 | `Q-common-1: ○○だと思う` | 問題に回答する |
| ヒント | `hint Q-common-1` | ヒントを表示（最大2回） |
| スキップ | `skip` | 次の問題へ |
| 解答表示 | `answer Q-common-1` | 解答を直接表示 |
| 進捗 | `status` | 回答状況を表示 |
| 質問 | `q: ○○について教えて` | プロジェクトへの自由質問 |

では共通の最初の問題から始めましょう。準備ができたら回答してください。
```

5. ユーザーが回答したら、**その問題に関連するプロジェクトのコードを実際に読みに行き**、回答の正確性を評価する。解答は事前に持っていないので、問題文とプロジェクトの実態を照合して判定する。

6. フィードバックは以下のフォーマットで返す:

```
### <問題ID>: <問題タイトル>

**判定**: ⭕ 正解 / 🔺 部分的に正解 / ❌ 不正解

**あなたの回答の評価**:
（回答の良かった点、足りなかった点を具体的に指摘）

**解説**:
（解説は以下の原則に従って書く）

**参照**: `ファイルパス:行番号`
```

---

## 解説の書き方（最重要）

newcomer はプロジェクトの文脈を持っていない。解説は「プロジェクトを知っている人向けの補足」ではなく「初めて見る人が理解できる説明」にする。

### 原則 1: まず「何の話か」から入る

いきなり詳細に入らず、その問題が扱うトピックの位置づけを示す。

悪い例:
> `middleware.ts` では `NextResponse.next()` を使ってリクエストを通過させています。

良い例:
> このプロジェクトでは、ユーザーのリクエストが各ページに届く前に、必ず通過する「門番」のような仕組みがあります。それが `middleware.ts` です。ここで「このユーザーはログイン済みか？」「この URL にアクセスする権限があるか？」を一括でチェックしています。

### 原則 2: 「なぜ」を必ず説明する

事実だけでなく、その設計判断の背景・動機を示す。

悪い例:
> Server Components を使っています。

良い例:
> このページでは Server Components を使っています。理由は、ユーザーがボタンをクリックしたりフォームに入力したりする操作がなく、データを表示するだけだからです。Server Components にすることで、ブラウザに送る JavaScript の量が減り、ページの表示が速くなります。

### 原則 3: 関連する全体像を添える

その知識がプロジェクトのどこに繋がるかを示す。

例:
> この認証の仕組みは、バックエンド側の `apps/api/src/routes/auth.ts` と対になっています。フロントエンドで JWT をヘッダーに付けて送り、バックエンドでそれを検証する流れです。

### 原則 4: Level に応じて深さを変える

- **Level 1（地図を読む）**: 観察した事実を丁寧に確認。「それが何のためのものか」を平易に説明する。専門用語には必ずインライン説明を添える
- **Level 2（つながりを理解する）**: 2つの事実の関係を説明。「○○があるから○○が可能になっている」のように因果を明示する
- **Level 3（設計意図を読む）**: 設計意図 + トレードオフ。「もし○○だったら」の対比で理解を深める
- **Level 4（パターンを活用する）**: パターンの背景 + 「こうしなかったらどうなるか」の具体例。コードレビュー問題では指摘の根拠を丁寧に
- **Level 5（設計シミュレーション）**: 複数アプローチを並べ、それぞれの長所・短所を比較。このプロジェクトでの推奨理由を明示する

### 原則 5: 専門用語にはインライン説明を付ける

newcomer が知らない可能性のある用語は、初出時にインラインで説明する:

> RSC（React Server Components: サーバー側でのみ実行されるコンポーネント）を使って...
> ORM（Object-Relational Mapping: DBのテーブルをプログラムのオブジェクトとして扱う仕組み）である Prisma が...

---

7. フィードバックの原則:
   - まず回答の良い点を認める（部分的にでも合っていれば明示する）
   - 足りない部分は「○○の観点が抜けている」のように具体的に
   - 正解でも「さらに深い理解」として関連知識を提供する
   - 必要に応じて実際のプロジェクトコードを引用する
   - 次の問題との関連がある場合はその橋渡しをする

8. コマンドの処理:
   - `hint <問題ID>` — 解答は出さず、どのファイルを読むべきか、何に注目すべきかのヒントを出す。2回目はより具体的に
   - `skip` — 解答を表示せず次の問題へ。「あとで戻っても OK」と伝える
   - `answer <問題ID>` — プロジェクトコードを読んで解答を直接表示する（解説の書き方原則に従う）
   - `status` — 領域ごと・Level ごとの回答状況を表示

## Q&A 機能

9. ユーザーが `q:` で始まるメッセージを送った場合、または明らかにレッスン問題への回答ではなくプロジェクトへの自由質問をしている場合、Q&Aモードとして処理する。

### Q&A の回答方法

- プロジェクトのコードを実際に読みに行き、正確な情報に基づいて回答する
- 該当するファイルパスと行番号を必ず示す
- **解説の書き方の原則をそのまま適用する**（「何の話か」から入り、「なぜ」を説明し、全体像を添える）

### Q&A のフィードバック蓄積

質問と回答のペアを `LESSON_FEEDBACK.md` に追記する。以下のフォーマットで蓄積する:

```markdown
### [Q&A] <質問の要約（1行）>

- **質問者**: newcomer
- **日時**: YYYY-MM-DD
- **関連**: <問題ID>（レッスン問題に関連する場合）/ なし
- **質問**: <質問の全文>
- **回答要約**: <回答の要点を2-3行で>
- **参照**: `ファイルパス:行番号`
- **提案**:
  - [ ] **LESSON**: <この質問から新しいレッスン問題を作るべきか、既存の問題を改善すべきかの提案。不要なら「なし」>
  - [ ] **CODE**: <コードやドキュメントにコメント追加・README改善等の提案。不要なら「なし」>
```

### Q&A 蓄積の判断基準

すべての Q&A を蓄積するのではなく、以下に該当するもののみ記録する:

1. **レッスンで問うべきだった知識**: 質問の内容がレッスンでカバーされていない重要な観点
2. **つまずきポイント**: 複数人が同じ箇所で困りそうな質問（コードが直感に反する、ドキュメント不足等）
3. **コード改善の種**: 「ここがわかりにくい」が、コード自体の改善機会を示唆している

蓄積しないもの:
- 一般的な技術知識の質問
- 個人的な環境構築トラブル
- 1回限りの単純な確認質問

### Q&A 蓄積時のユーザーへの表示

フィードバックを蓄積した場合、回答の末尾に以下を追加する:

```
> 📝 この Q&A を LESSON_FEEDBACK.md に記録しました。
> 提案: [LESSON: ○○ / CODE: ○○]
```

蓄積しなかった場合は何も表示しない。

## LESSON_FEEDBACK.md のフォーマット

初回作成時、以下のヘッダーを付ける:

```markdown
# Lesson Feedback

> このファイルは `/start-lesson` 実行時に自動生成されます。

>
> **リードへ**: 各提案の `[ ]` チェックボックスを確認し、
> 対応したものはチェックを入れてください。

---

## Q&A ログ

（ここに Q&A が追記される）

---

## レッスン結果サマリー

（レッスン完了時に追記される）
```

## レッスン完了時

10. 全問完了（skip含む）したら、以下のサマリーを出力する:

```
## レッスン完了

### スコア

| 領域 | Level | 正解 | 部分正解 | 不正解 | スキップ |
|---|---|---|---|---|---|
| 共通 | L1 | x | x | x | x |
| 共通 | L2 | x | x | x | x |
| ... | ... | ... | ... | ... | ... |

### 強み
（よくできていた観点）

### 伸びしろ
（理解が浅かった観点 + 深めるために読むべきファイルやドキュメント。
 解説の書き方原則に従い、「何を読むべきか」だけでなく「なぜそれを読むと理解が深まるか」も説明する）

### 次のステップ
（このプロジェクトで最初に取り組むのにおすすめのタスク。
 回答傾向から推測した newcomer の強みを活かせるタスクを提案する）
```

11. さらに `LESSON_FEEDBACK.md` のレッスン結果サマリーセクションに以下を追記する:

```markdown
### レッスン結果: YYYY-MM-DD

**スコア**: ⭕ x / 🔺 x / ❌ x / スキップ x（全 N 問）

**つまずいた問題**:
- <問題ID>: <問題タイトル> — <何が足りなかったかの要約>
- ...

**提案**:
- [ ] **LESSON**: <正答率が低い問題の改善提案（問題文がわかりにくい？前提知識が足りない？ヒントを追加すべき？）>
- [ ] **CODE**: <つまずきの原因がコードの複雑さやドキュメント不足にある場合の改善提案>
```
