プロジェクトの資料・コード・設定を探索し、newcomer 向けの問題形式レッスン（LESSON.md）を生成する。
生成されたファイルはコミットして共有する。newcomer は `/start-lesson` でレッスンを開始する。

## 目的

AIがコードを書く時代において、開発者に求められるのは:

- **セルフレビュー力**: AI が生成したコードがプロジェクトの規約・設計方針に沿っているか判断できる
- **設計判断力**: 「なぜこうなっているか」を理解し、新機能の実装方針を適切に決められる
- **全体像の把握**: 変更が他のどこに影響するかを見通せる

問うべきは「コードの書き方」ではなく「コーディングの意思決定」。
newcomer が自分の担当領域で自律的に判断できるようになることがゴール。

## Workflow

### Phase 1: プロジェクト探索（並列実行）

カレントディレクトリを起点に、以下の情報を **並列** で収集する。

#### 1a. ドキュメント検出

以下のファイルを探索し、存在するものをすべて読み込む:

```
README.md, README.rst, README.txt
CONTRIBUTING.md, CONTRIBUTING.rst
ARCHITECTURE.md, DESIGN.md, OVERVIEW.md
docs/**, wiki/**
CLAUDE.md, .claude/CLAUDE.md
CHANGELOG.md, HISTORY.md
```

#### 1b. プロジェクト設定・メタデータ

以下から技術スタック・依存関係を特定する:

```
package.json, pnpm-lock.yaml, bun.lockb, yarn.lock
pyproject.toml, requirements.txt, Pipfile, uv.lock
Cargo.toml, go.mod, Gemfile, build.gradle, pom.xml
composer.json, mix.exs, pubspec.yaml
```

#### 1c. インフラ・CI/CD 設定

```
Dockerfile, docker-compose.yml, docker-compose.yaml
.github/workflows/**, .gitlab-ci.yml, Jenkinsfile
vercel.json, netlify.toml, wrangler.toml, wrangler.jsonc
fly.toml, render.yaml, railway.json
```

#### 1d. コード構造の把握

```bash
find . -maxdepth 3 -type d \
  ! -path '*/node_modules/*' ! -path '*/.git/*' \
  ! -path '*/dist/*' ! -path '*/.next/*' \
  ! -path '*/__pycache__/*' ! -path '*/target/*' \
  ! -path '*/.venv/*' ! -path '*/vendor/*' \
  | head -100
```

#### 1e. Git 情報（リポジトリの場合）

```bash
git remote -v 2>/dev/null
git log --oneline -20 2>/dev/null
```

### Phase 2: ドキュメント内リンク追跡

Phase 1 で収集したドキュメント内の Markdown リンク `[text](path)` を解析し、
プロジェクト内ファイルを参照しているものを追跡・読み込む。
外部 URL は参考リンクとして記録のみする。

### Phase 3: コードベース深堀り

Phase 1-2 の結果を元に、出題に必要な以下を重点的に読み込む:

1. **エントリーポイント**: `src/index.ts`, `app/layout.tsx`, `main.py`, `cmd/main.go` 等
2. **ルーティング/API定義**: `app/` 構造、`routes/`, API ハンドラー等
3. **型定義・モデル**: `types/`, `models/`, `schema/`, `prisma/schema.prisma` 等
4. **設定ファイル**: `tsconfig.json`, `biome.json`, `vite.config.ts` 等
5. **テスト**: 代表的なテストファイルを2-3個（パターン把握）
6. **設計パターン**: DI、リポジトリパターン、状態管理等
7. **環境変数**: `.env.example` 等（`.env` 本体は絶対に読まない）

### Phase 4: ドメイン分析とグルーピング

**Phase 3 の後、問題を作る前に**、プロジェクトのドメイン構造を分析する。

#### 4a. ドメイン領域の特定

プロジェクトのディレクトリ構造・コード・設定から、意味のある領域を特定する。
newcomer がフルスタックとは限らないため、領域ごとに独立して取り組めるようにする。

典型的な分け方の例:

| プロジェクト構成 | 領域の分け方 |
|---|---|
| モノレポ (apps/ + packages/) | アプリ単位 + 共有パッケージ |
| フルスタック (frontend + backend) | フロントエンド / バックエンド / インフラ |
| バックエンド API のみ | ドメインモデル / API レイヤー / インフラ |
| ライブラリ | Core API / プラグイン / ビルド・配布 |

**重要**: 上記は例示。実際のプロジェクト構造に合わせて領域名を決める。
`apps/web`, `apps/api`, `packages/ui` のようなディレクトリ名をそのまま領域名にしてもよい。

#### 4b. 領域ごとの出題計画

各領域に対して、Level 1-5 の問題を配分する。
全領域に共通する問題（プロジェクト全体像、共有コード）は「共通」領域としてまとめる。

### Phase 5: LESSON.md 生成

探索結果と領域分析をもとにレッスンを生成し、カレントディレクトリに `LESSON.md` として書き出す。

## 問題 ID の命名規則

- `Q-common-1`, `Q-common-2`, ... — 共通問題
- `Q-frontend-1`, `Q-frontend-2`, ... — フロントエンド領域
- `Q-api-1`, `Q-api-2`, ... — API 領域
- 領域名はプロジェクト構造に合わせて自由に決める
- 各領域内で Level 1 → 5 の順に番号を振る（Level をまたいでも通し番号）

## 難易度グラデーション

### Level ごとの役割

| Level | 問う力 | 問題の性質 | 回答の形 |
|---|---|---|---|
| **1: 地図を読む** | 「どこに何があるか」を知る | 読む対象を指定し、事実を読み取らせる | 事実ベース（ファイル名、ツール名、設定値） |
| **2: つながりを理解する** | 「これとあれはどう関係するか」を読む | Level 1 で学んだ事実同士の関係を問う | 関係の説明（「○○は○○のために○○している」） |
| **3: 設計意図を読む** | 「なぜこうなっているか」を考える | 既存の設計判断の背景・意図を問う | 理由の説明 + 代替案との比較 |
| **4: パターンを活用する** | 「このプロジェクトではこうする」を掴む | 暗黙の規約やパターンの認識・適用を問う | パターンの言語化 or コードレビュー |
| **5: 設計をシミュレーション** | 「新しいものをどう作るか」を決められる | 正解が1つでない設計判断を問う | 方針 + トレードオフの考察 |

### Level 1: 問題文の書き方

Level 1 は newcomer が **最初に触れる** 問題。前提知識ゼロで取り組めるように、
問題文自体が「何を見ればいいか」をガイドする。

#### Level 1 の原則: 読む対象を指定し、観察を促す

**悪い例（前提知識を要求してしまう）**:
> turbo.json のパイプライン設定を説明せよ

→ newcomer は turbo.json が何かも、パイプラインが何を意味するかもわからない

**良い例（読む対象を指定し、観察を促す）**:
> このプロジェクトのルートに `turbo.json` というファイルがあります。
> これはモノレポ（複数のアプリやライブラリを1つのリポジトリで管理する構成）の
> ビルドツール Turborepo の設定ファイルです。
>
> `turbo.json` を開いて、以下を読み取ってください:
> 1. どんなタスク（build, test, lint 等）が定義されているか
> 2. タスク間に依存関係（`dependsOn`）は設定されているか
> 3. どのタスクにキャッシュ（`cache`）が効いているか

#### Level 1 の問題文テンプレート

```
<対象ファイル/ディレクトリの紹介: 何のためのものか、1-2文で>

<そのファイルを開いて、以下を読み取ってください:>
1. <具体的な観察ポイント1>
2. <具体的な観察ポイント2>
3. <具体的な観察ポイント3（オプション）>
```

Level 1 前半は1つのファイルから、後半は2-3のファイルを照合する構成にする。

### Level 2: 問題文の書き方

Level 2 は Level 1 で観察した事実を「つなげる」問題。
問題文で **Level 1 の問題を参照** し、その知識を前提にする。

**良い例**:
> Q-common-1 で `turbo.json` のタスク定義を確認しました。
> 次に `package.json` の `scripts` セクションと見比べてみてください。
>
> - `turbo.json` で定義されたタスクと `package.json` の scripts はどう対応していますか？
> - 開発者が `pnpm build` を実行したとき、実際には何が起きますか？
>   各パッケージの build がどの順序で実行されるか、turbo.json の設定から推測してください。

#### Level 2 の原則: Level 1 の知識を橋渡しする

- 問題文の冒頭で「Q-xxx-N で ○○ を確認しました」と前の問題を参照する
- 新しいファイルやコンセプトが登場する場合は、Level 1 と同様に簡潔に紹介する
- 「比較してください」「対応を見つけてください」のように、2つの事実を結びつける問いにする

### Level 3: 問題文の書き方

Level 3 は「なぜ」を問う。Level 1-2 で掴んだ事実と関係をもとに、設計判断の意図を考えさせる。

**良い例**:
> Q-common-1, Q-common-2 で、このモノレポでは Turborepo でタスクを管理し、
> パッケージ間に依存関係があることを確認しました。
>
> ここで `packages/` ディレクトリを見ると、`packages/ui` と `packages/config` があります。
> `packages/ui` は `packages/config` に依存していますが、逆方向の依存はありません。
>
> - この「依存の方向」はなぜ一方向なのでしょうか？
> - もし双方向に依存させたら、どんな問題が起きると思いますか？

#### Level 3 の原則: 事実を示した上で「なぜ」を問う

- 問う対象の事実を問題文の中で提示する（newcomer に発見させるのではなく、考察に集中させる）
- 「もし○○だったら」の反実仮想を含めると、設計意図がより明確になる
- ここで初めて「推察せよ」「考えてみてください」のような問い方を使う

### Level 4: 問題文の書き方

Level 4 は既存のパターンを認識し、適用する問題。

**良い例**:
> このプロジェクトの `apps/web/src/components/` にあるコンポーネントを3-4個見比べてみてください。
>
> - ファイルの構成（何がエクスポートされているか、テストファイルの有無）に共通するパターンはありますか？
> - 新しいコンポーネント `UserAvatar` を作るとしたら、このパターンに従うとどういうファイル構成になりますか？

コードレビュー問題もこの Level に配置する:
> 以下のコードは、このプロジェクトに新しく追加される予定のコンポーネントです。
> これまでに学んだこのプロジェクトのパターンに照らして、改善すべき点を指摘してください。

### Level 5: 問題文の書き方

Level 5 は正解が1つでない設計判断。十分な文脈を問題文で与える。

**良い例**:
> このプロジェクトに「ユーザーが自分のプロフィール画像をアップロードできる機能」を追加することになりました。
>
> 現在の技術スタックと既存の設計パターンを踏まえて、以下を検討してください:
> - 画像はどこに保存しますか？（選択肢: DB に blob / オブジェクトストレージ / ローカルファイルシステム）
> - アップロード API はどこに配置しますか？（既存の API パターンを参考に）
> - フロントエンドのフォームはどの既存ページを参考に作りますか？
>
> それぞれの判断について、このプロジェクトの既存パターンとの整合性を考慮してください。

### 全 Level 共通: 問題文に含めるべきもの

| 要素 | Level 1-2 | Level 3-4 | Level 5 |
|---|---|---|---|
| 対象ファイルの紹介 | 必須（何のファイルか説明） | 必要に応じて | 不要（既知の前提） |
| 専門用語の説明 | 必須（初出時にインライン説明） | 初出のみ | 不要（既知の前提） |
| 前の問題への参照 | 不要（最初の問題なので） | 推奨（Level 1-2 を参照） | 推奨（全体を振り返る） |
| 具体的な観察指示 | 必須（「○○を開いて」） | 任意 | 不要 |
| 「なぜ」の問い | 不要（事実の読み取りに集中） | 必須 | 必須 |
| 反実仮想 | 不要 | 推奨（「もし○○だったら」） | 推奨 |

### 問題間の連鎖設計

Level 1 → 5 は独立した問題の羅列ではなく、**前の問題で得た知識が次の問題の土台になる** 構造にする:

```
Level 1: turbo.json を読む → タスク定義を知る
    ↓ この知識が前提
Level 2: package.json と照合 → ビルドの流れを理解する
    ↓ この知識が前提
Level 3: 依存の方向が一方向な理由を考える
    ↓ この知識が前提
Level 4: 新しいパッケージを追加するときの規約を見抜く
    ↓ この知識が前提
Level 5: 既存パッケージの分割を設計する
```

問題文の冒頭で「Q-xxx-N で○○を確認しました」と明示的に参照することで、
newcomer が Level を順に追えば自然に知識が積み上がるようにする。

## 問題設計の原則

### 問うべきこと: 意思決定とプロジェクト文脈

問題は **コーディングの意思決定** を問う。コードの書き方や行数の指摘ではない。

**良い問い方**:
- 「認証ミドルウェアがこの層に配置されている理由を、リクエストの流れと関連づけて説明せよ」
- 「新しいページを追加するとき、このプロジェクトではどの既存ページを参考にすべきか。その理由も述べよ」
- 「この API エンドポイントのレスポンス型が共通型を使わず独自に定義されている。この判断の妥当性を評価せよ」

**悪い問い方（避けるべき）**:
- 「`src/lib/auth.ts` の 42 行目で何をしているか説明せよ」（← コードの読み上げ）
- 「このファイルの行数を答えよ」（← 意味のないトリビア）
- 「○○関数のパラメータの型を答えよ」（← IDE で見ればわかる）

### プロジェクト全体を考慮した問い

問題は常に **プロジェクト全体の中での位置づけ** を意識させる:

- 「このコンポーネントは他のどこから使われているか、そしてなぜ共有コンポーネントとして切り出されているか」
- 「このバリデーションがフロントエンドとバックエンドの両方に存在する理由」
- 「この設定変更がCI/CDパイプラインに与える影響」

### 問題のアンチパターン

- 「○○とは何ですか？」のような用語定義（→ 用語集に載せる）
- 一般的な技術知識で答えられる問題（→ プロジェクト固有の文脈を必ず含める）
- Yes/No で答えられる問題
- 答えが1単語で済む問題
- 特定の行番号やコード片を暗記させる問題

## Level 4: コードレビュー問題の作り方

1. プロジェクト内の実際のコードから **正しいパターン** を2-3個特定する
2. そのパターンに **意図的に違反した** コード片を作成する（実際のコードは改変しない）
3. 違反は **設計判断レベル** のものにする:
   - 良い例: 責務の配置ミス、不適切なレイヤー間依存、パターン不一致
   - 悪い例: インデントの乱れ、変数名のタイポ、import 順序
4. 違反は1問につき2-4個含める（多すぎると焦点がぼやける）

## 出力ルール

1. **LESSON.md には問題文のみ**: 解答・ヒント・参照先は一切書かない
2. **問題数**: 「共通」5-8問 + 各領域 5-10問 程度。領域数に応じて調整
3. **領域ごとに独立**: ある領域の問題を解くのに、別領域の知識を前提としない
4. **用語集を充実させる**: 問題文で使うドメイン用語・略語はすべて付録の用語集に含める
5. **秘密情報**: API キー、トークン、パスワード等は絶対に含めない
6. **`.env` 本体は絶対に読まない**
7. **完了報告**: 出力後、領域ごとの問題数・Level分布・主なトピックを報告する
8. **コミット案内**: 「`LESSON.md` をコミットしてリポジトリに含めてください。newcomer は `/start-lesson` で開始できます」と案内する

## 補足: 技術スタック別の出題ポイント

### Next.js (App Router)
- Server Components と Client Components の境界はどこで、なぜそこで分けているか
- データフェッチをどこで行うべきかの判断基準
- キャッシュ戦略がユーザー体験にどう影響するか
- middleware が担う責務と、そこで処理すべきでないこと

### Python (FastAPI/Django)
- ドメインモデルと API スキーマを分けている理由
