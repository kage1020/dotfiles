# Diff Report

フィーチャーブランチの変更内容を分析し、構造化されたレポートを会話内に Markdown で出力する。

## Workflow

### Phase 1: 環境検出

1. **ベースブランチ決定**: 引数があればそれを使用。なければ以下の順で自動検出:
   ```bash
   git rev-parse --verify main 2>/dev/null && echo main ||
   git rev-parse --verify dev 2>/dev/null && echo dev ||
   git rev-parse --verify master 2>/dev/null && echo master
   ```
2. **分岐点特定**:
   ```bash
   MERGE_BASE=$(git merge-base <base-branch> HEAD)
   ```
3. 現在のブランチがベースブランチそのものなら「差分なし」と報告して終了。

### Phase 2: データ収集（並列実行）

以下のコマンドを **並列** で実行する:

```bash
# 統計情報
git diff --stat $MERGE_BASE..HEAD
git diff --numstat $MERGE_BASE..HEAD
git diff --name-status $MERGE_BASE..HEAD

# コミット履歴
git log --oneline $MERGE_BASE..HEAD

# 差分サイズ確認（行数カウント）
git diff $MERGE_BASE..HEAD | wc -l
```

### Phase 3: 差分サイズ判定と読み込み戦略

差分の総行数に応じて読み込み戦略を切り替える:

| 規模 | 行数 | 戦略 |
|---|---|---|
| 小規模 | ~1000行 | `git diff $MERGE_BASE..HEAD` 全体を精読 |
| 中規模 | 1000~5000行 | ファイル単位で `git diff $MERGE_BASE..HEAD -- <file>` を優先度順に読み込み。優先度: 新規追加 > 変更量大 > テスト > 設定ファイル |
| 大規模 | 5000行~ | 統計ベースで全体像を把握 + 変更量上位10ファイルを精読 |

### Phase 4: グルーピング

変更ファイルを論理的な機能単位にグループ化する。以下の観点で判断:

- **コンポーネント単位**: 実装 + テスト + スタイル（例: `Button.tsx` + `Button.test.tsx` + `Button.module.css`）
- **機能単位**: 同一ディレクトリ内の関連ファイル、同一機能に属するファイル群
- **レイヤー単位**: API route + service + repository + 型定義など、同一機能の異なるレイヤー
- **インフラ/設定**: 設定ファイル、CI/CD、依存関係の変更

グループ名は変更内容の意図を反映した名前にする（ファイル名やディレクトリ名ではなく）。

### Phase 5: レポート生成

以下の5セクションを Markdown で出力する。レポートの前後に余計な説明を付けない。

## レポートフォーマット

```
## Diff Report: <current-branch> (vs <base-branch>)

### 1. 変更統計

| 指標 | 値 |
|---|---|
| コミット数 | N |
| 変更ファイル数 | N |
| 追加行数 | +N |
| 削除行数 | -N |

**ファイル種別内訳**:
- TypeScript/TSX: N files
- CSS/SCSS: N files
- ...

---

### 2. 機能サマリー

変更を論理的な機能単位でグループ化。

#### <機能グループ名>
<概要: 何を実装/変更したか 1-2文>

対象ファイル:
- `path/to/file.ts` (A/M/D) — 役割の簡潔な説明
- ...

#### <機能グループ名>
...

---

### 3. 技術ポイント

実装における注目すべき技術的判断・パターン。

- **<ポイント名>**: 説明。なぜこの判断をしたか、どういうトレードオフがあるか。
  - 該当: `path/to/file.ts:L行`
- ...

---

### 4. 理解度チェック

コードの品質・明瞭性に関する指摘。以下の観点で検出する:

- マジックナンバー / ハードコード値
- 複雑なロジック（ネストが深い、条件分岐が多い）
- 異常パターン（通常と異なる書き方、意図不明なコード）
- 潜在バグ（エッジケース未処理、off-by-one、null チェック漏れ）
- エラーハンドリング不足（catch の握りつぶし、ユーザーへの技術エラー露出）
- 不明瞭な命名（略語、汎用的すぎる名前、誤解を招く名前）

各指摘のフォーマット:
- **[カテゴリ]** `file.ts:L行` — 具体的な指摘内容

指摘がなければ「特筆すべき問題なし」と記載。無理に指摘を作らない。

---

### 5. コミット履歴

| Hash | Message |
|---|---|
| `abc1234` | feat: ... |
| ... | ... |
```

## 注意事項

- レポートの前後に余計な説明や前置きを付けない。レポートだけを出力する。
- 差分が空（コミットなし）の場合は「ベースブランチとの差分はありません」とだけ出力する。
- 理解度チェックは厳しく行う。「動くからOK」で見逃さない。ただし無理に指摘を作らない。
- 技術ポイントは実装者の意図を汲み取り、良い判断も悪い判断も両方記載する。
- 大規模差分の場合でも、各セクションの情報密度を維持する。冗長にならないよう要約する。
