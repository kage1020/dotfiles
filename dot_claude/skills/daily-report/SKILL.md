# Daily Report

当日の全作業ログを集約し、不足情報を補完した上で、フォーマットに沿った日報を生成する。

## Workflow

### Phase 1: ログ集約

1. 集約スクリプトを実行して当日のログデータを取得する:

   ```bash
   python3 ~/.claude/hooks/aggregate-daily-log.py --date "$(date -u +%Y-%m-%d)"
   ```

   引数で日付を指定された場合はその日付を使用する（例: `/daily-report 2026-04-08`）。

2. 出力された JSON からセッション情報・ツール使用ログ・メモ・git コミットを把握する。

### Phase 2: トランスクリプト分析

1. 集約結果の `projects[].sessions[].session_id` と `timeline` を元に、主要な作業セッションを特定する。

2. 各プロジェクトの当日のトランスクリプトファイルが存在すれば、assistant の発言から以下を抽出する:
   - 重要な設計判断とその理由
   - 遭遇した問題と解決策
   - 中断された作業や未完了タスク

   トランスクリプトのパスは `~/.claude/projects/{PROJECT_SLUG}/` 配下にある。
   ファイルが大きい場合は末尾 2000 行を対象にする。

### Phase 3: 情報補完（対話）

集約したログから日報に必要な情報が不足している場合、ユーザーに質問する。
以下の観点で不足を判定する:

- **作業の目的・背景**: ログからはツール操作しか分からない場合、「なぜその作業をしたか」を確認する
- **成果の評価**: 作業が完了したのか途中なのか、期待通りの結果だったか
- **課題・ブロッカー**: セッションが短時間で終了している場合や、同じ操作を繰り返している場合
- **翌日の予定**: ログからは推測できないため、必ず確認する
- **共有事項**: チームに共有すべきことがあるか

質問は一度にまとめて行う（何度も聞き返さない）。
ログから十分に情報が得られている場合は質問を最小限にする。

### Phase 4: 日報生成

収集した全情報を元に、以下のフォーマットで日報を生成する。
生成した日報は `~/.claude/reports/daily-report-{DATE}.md` に保存する（ディレクトリがなければ作成する）。

## 日報フォーマット

```markdown
# 日報 {YYYY-MM-DD}

## 作業サマリー

| 指標 | 値 |
|---|---|
| 作業時間 | Xh Xm |
| セッション数 | N |
| コミット数 | N |
| 主要プロジェクト | project-name |

## 作業内容

### {プロジェクト名 or タスク名}

**目的**: なぜこの作業をしたか

**実施内容**:
- 具体的な作業内容（ツールログ・コミットから再構成）
- ...

**成果**:
- 達成したこと
- 作成・変更したファイルや機能

**思考・判断の記録**:
- 作業中に考えたこと、判断した理由（メモやトランスクリプトから抽出）
- ...

### {プロジェクト名 or タスク名}
...

## ターミナル実行ログ（主要）

作業中に実行した主要なコマンドを時系列で記載する。
全てのコマンドを列挙するのではなく、作業の流れが分かる重要なものだけを抽出する。

| 時刻 | コマンド | 目的 |
|---|---|---|
| HH:MM | `command` | なぜ実行したか |
| ... | ... | ... |

## AI Agent 操作ログ

Agent ツールの使用履歴を記載する。使用がなければこのセクションは省略する。

| 時刻 | エージェント種別 | 目的 |
|---|---|---|
| HH:MM | subagent_type | description |
| ... | ... | ... |

## 課題・問題点

- 遭遇した問題とその状態（解決済み / 未解決）
- ブロッカーがあれば記載

## 翌日の予定

- 明日やること
- 優先度が高いもの

## 所感

作業を通じて感じたこと、気づき、改善点など。
```

## 注意事項

- タイムスタンプは UTC で記録されているため、日報の時刻表記はローカルタイム（JST = UTC+9）に変換する。
- ログが空（セッションなし、メモなし、コミットなし）の日は「作業記録なし」とだけ出力する。
- メモ（`/memo`）の内容は「思考・判断の記録」セクションに統合する。
- ツール操作ログの羅列ではなく、**何をしたか・なぜしたか** の文脈で再構成する。
- git コミットメッセージは作業内容の裏付けとして使う。コミットの羅列はしない。
- 日報の前後に余計な説明を付けない。日報のみを出力してファイルに保存する。
- 情報が豊富な日は詳細に、少ない日は簡潔に。情報密度を一定に保つ。
