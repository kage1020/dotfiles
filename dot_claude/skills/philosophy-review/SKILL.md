# Philosophy Review

コード変更を、以下の6つの原則に照らしてレビューし、違反・懸念・称賛をレポートする。

## レビュー手順

### Phase 1: 静的レビュー（コードリーディング）

1. `git diff` で変更差分を取得する（ステージ済み + 未ステージ）
2. 差分が空なら、直近のコミットの差分を対象にする
3. 変更ファイルの全体コンテキストも必要に応じて読む（差分だけでは判断できない場合）
4. 6つの原則に照らしてコードを静的にレビューする

### Phase 2: 動的レビュー（ブラウザ実機確認）

静的レビューで UI/UX に関する指摘が出た場合、コードを読んだだけでは確信が持てない場合、または変更が UI コンポーネントに関わる場合は、実際にブラウザで動作を確認する。コードの見た目と実際のユーザー体験は異なることが多いため、時間を惜しまず確認する。

手順:

1. dev server が起動していなければ起動する（`package.json` の `scripts` を確認）
2. playwright-cli または agent-browser スキルを使って該当ページにアクセスする
3. 以下の観点で実際の操作を行い確認する:
   - モーダルやダイアログが表示されるタイミングと頻度
   - disabled 要素の存在と操作フローへの影響
   - エラー発生時に実際に表示されるメッセージ
   - キーボード操作（Tab 移動、Enter、Esc、ショートカット）の挙動
   - スクリーンリーダーで読み上げた際の体験（aria 属性の実効性）
   - ローディング中の操作可否
   - レスポンシブ対応（モバイル幅での表示）
4. スクリーンショットを撮り、視覚的に問題がないか確認する
5. 静的レビューの指摘が実際の体験と一致するか照合し、レポートに反映する

動的レビューで新たに発見した問題は、静的レビューの結果に追記する。動的レビューで確認した項目はレポートの総評に「ブラウザ確認済み」と明記する。

## 6つの原則

### UXの3制約

#### 1. ユーザーの意図を裏切らない

ユーザーは意識的・無意識的に意図を持って操作する。直感的・慣習的な期待を逸脱する仕様は第1級の不快感を与える。アクセシビリティもこの範疇。

チェック観点:

- リンクやボタンの見た目と実際の挙動が一致しているか（`<a>` が POST を飛ばす、ボタンがページ遷移するなど）
- キーボードショートカットが慣習に反していないか（例: Ctrl+S に「保存」以外の機能を割り当てるのは重大な違反）
- フォーム送信後の挙動がユーザーの期待通りか（送信したのに画面が変わらない、入力が消えるなど）
- WAI-ARIA、セマンティック HTML が正しく使われているか
- キーボード操作、スクリーンリーダーで意図通りに動くか
- 慣習的な UI パターンからの逸脱がないか（戻るボタンが進む、×ボタンが保存するなど）
- `preventDefault` や `stopPropagation` がブラウザのデフォルト挙動を不必要に殺していないか

#### 2. ユーザーの操作を邪魔しない

ユーザーは広告を見たいわけでもサービスを理解したいわけでもない。モーダル、スナックバー、disabled 要素はユーザーの操作への割り込み。操作は最大限受け入れ、自由な動線を確保する。

チェック観点:

- モーダルやダイアログが本当に必要か（インラインで解決できないか）。特に広告やお知らせをモーダルで画面上に被せる実装は重大な違反
- `disabled` 属性の使用に正当な理由があるか（操作を受け付けた上でバリデーションする方が良くないか）
- ローディング中にユーザー操作をブロックしていないか（楽観的更新は検討したか）
- 確認ダイアログが過剰でないか（Undo で代替できないか）
- トーストやスナックバーが操作の邪魔になっていないか
- フォーカストラップが適切か（必要なときだけか）

#### 3. ユーザーに技術的制約を強要しない

DB の atomic 制約、client/backend の同期、技術スタックの制限はサービス提供と無関係。それをユーザーに押し付けるのは怠慢。制約が必要ならサービス設計が間違っている。

チェック観点:

- エラーメッセージが技術用語になっていないか（「500 Internal Server Error」「Unique constraint violation」など）
- DB やAPI の制約がそのまま UI に露出していないか
- ネットワークエラーやタイムアウトをユーザーに直接見せていないか（リトライやフォールバックがあるか）
- ブラウザ/OS/フレームワーク/DB/インフラの制約をユーザーに転嫁していないか（ファイルサイズ制限、レスポンス遅延など。技術的に回避可能なら回避すべき）
- バリデーションルールが技術的制約の押し付けになっていないか（文字数制限の理由がDB カラム長など）

### エンジニアの3原則

#### 4. 足らぬなら作ってしまえホトトギス

既存ライブラリや API に完璧なものはない。不足があるなら自作する姿勢。

チェック観点:

- ライブラリの制限を「仕方ない」で受け入れていないか
- workaround がハック的になっていないか（自作した方がクリーンではないか）
- 「このライブラリではできない」で思考停止していないか

#### 5. 当たり前を疑う

インターネットの慣習は先人が構築した前提。必ずしも最適とは限らない。

チェック観点:

- 「みんなこうしているから」で実装していないか
- より良い UX パターンが存在しないか検討したか
- 既存の UI 慣習に縛られすぎていないか

#### 6. 開発者の当たり前 ≠ ユーザーの直感的

エンジニアは技術的最適に引っ張られがち。アプリを使うのはユーザーであり、開発者の意図とは異なる使い方をする。

チェック観点:

- 開発者にとって便利なだけの設計になっていないか
- ユーザーが「なぜこうなる？」と感じる箇所がないか
- テクニカルな知識がないと使えない UI になっていないか
- エラーハンドリングが開発者目線になっていないか

## レポート形式

以下の形式でターミナルに出力する。簡潔に、本質だけを伝える。

```
## Philosophy Review

### 違反 (Violations)
重大な原則違反。修正すべき。

- **[原則名]** ファイル:行番号
  何が問題で、なぜ問題か。改善案。

### 懸念 (Concerns)
原則に抵触する可能性。検討の余地あり。

- **[原則名]** ファイル:行番号
  懸念点と代替案。

### 詰めの甘さ (Halfway)
原則を理解していればそこで止まらないはず、という箇所。
違反ではないが、原則への理解が浅い・徹底されていない実装。
「動くからOK」で止まっている部分、一貫性の欠如、やりかけの抽象化など。

- **[原則名]** ファイル:行番号
  何が足りないか、原則を徹底するならどこまでやるべきか。

### 称賛 (Good)
原則を体現している実装。（あれば）

- **[原則名]** ファイル:行番号
  何が良いか。

### 総評
全体的な所感を1-2文で。
```

注意点:

- 違反も懸念もなければ「問題なし」とだけ出力する。無理に指摘を作らない。
- 各指摘には必ず「なぜそれが原則に反するか」の理由を含める。
- 改善案は具体的なコード例を含める（可能な場合）。
- 「詰めの甘さ」は特に重要なカテゴリ。原則の表面だけなぞった実装を見抜く。例えば:
  - 楽観的更新を実装したが失敗時のロールバックが一部だけ（一貫性の欠如）
  - Undo を実装したが一部の操作だけ対応（やりかけ）
  - エラーメッセージを抽象化したが、一部の経路では生のエラーが漏れている
  - アクセシビリティを意識しているが `aria-label` が一部欠けている
    こういった「方向性は正しいが徹底されていない」実装を指摘する。
- レポートの前後に余計な説明を付けない。レポートだけを出力する。
